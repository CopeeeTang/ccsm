# CCSM TUI 视觉与交互重构指南 (v3.0)

> **修订日期**: 2026-04-03
> **核心主题**: 极致减法、Spine View (脊索结构) 与抽屉式详情展示
> **前置依赖**: 本设计用于替代先前的平铺式三栏布局以及琐碎的 SessionGraph/Detail 区块。

---

## 一、 核心设计理念

针对原来使用时“看一眼眼花缭乱”、“信息无焦点”、“难以快速定位进行中任务”的痛点，新版 TUI 前端的设计理念是：**做减法、重构主干、焦点隔离**。

1. **做减法：化常驻为按需弹窗**
   - 彻底废弃原本 `[ 左 20% ] [ 中 40% ] [ 右 40% ]` 的三面板平铺切割。
   - 默认展示为 **两栏结构**：右侧省下的空间全部归还给中间栏，让会话标题和摘要得到舒展展示。右侧详情页转化为**右侧滑出抽屉 (Drawer)**，只有明确需要看上下文时才会调出。
2. **重构主干：融合时间线与节点血缘**
   - 原先仅仅依赖时间排序，或是由于 Compact / Fork 规则而产生无限嵌套树。
   - 新版引入 **Spine View (脊索骨干图)**，打造出类似于 GitHub PR 演进或者 Vercel 部署轨道的直观视觉体系。
3. **焦点隔离：详情信息块重组**
   - 抽屉内的 Detail 要告别满屏黑白文本的压迫感，采用四大不同风格的组件进行职能隔离。

## 二、 布局响应规则 (Architecture)

```text
┌────────────────────────────────────────────────────────┐
│ ⬡ CCSM | The Ultimate Spine View                       │
╞══════════════╤═════════════════════════════════════════╡
│ 📁 Worktrees │ 🚀 TUI 详情及组件优化 [Active Workflow]    │
│              │                                         │
│ ▸ feature/A  │ 10:00 ┃  ⬤ [Compact] Drawer UI 开发中    │
│   main       │       ┃    💬 正在调整 Drawer 焦点逻辑... │
│   fix/bug_1  │       ┃                                 │
│              │       ┃                                 │
│              │ 09:15 ┣━▶ ◿ [Fork] 用 React 写前端       │
│              │       ┃    💭 实验并未成功，废弃         │
│              │       ┃                                 │
│              │ 08:00 ┃  ◖ [Compact] TUI 框架搭建        │
└──────────────┴───────┻─────────────────────────────────┘
                       ⬇ 点击卡片后，呼出 Drawer Overlay ⬇
               ┌─────────────────────────────────────────┐
               │ ▉▉▉▉▉▉▉▉▉ Drawer Panel (60% Width)      │
               └─────────────────────────────────────────┘
```

**交互机制：**
- **默认闭合**: `app-container` 维持完整的 `20:80` 面板比例分配。
- **触发抽屉**: 选中某个 Session Card 按下 `Enter`，或点击产生焦点时，呼叫 `Drawer` 滑出。
- **遮罩行为**: 抽屉滑出时伴随全屏半透明遮罩背景框（暗色调，微弱模糊），从而在视觉焦点上完全切开左中两侧的信息。
- **退出交互**: 全局监听 `Esc` 或专门的右上角 `x` 关闭抽屉，释放屏幕焦点。

## 三、 中栏设计：The Spine View (脊索结构图)

Spine View 旨在优雅结合“物理向下流动的时间序列”和“节点间的 Fork/Compact 血缘关系”。

### 3.1 渲染结构
1. **时间主列 (Left-most)**:
   - 固宽列（例如 100px）。最左侧展示该节点的 `last_message_at` 的人类易读版（如：今天 10:00，昨天 18:30）。
2. **连线枢纽区 (Center Spine)**:
   - 这是一个独立的视觉绘画区。
   - **Compact (继承链)**: 在具有 Compact 父子关系的多个节点间，中间穿插一根醒目的、加粗带色彩的主垂直连线（`u2503` ┃）。这代表一个核心的“主干任务流”。
   - **Fork (分叉链)**: 若某节点属于 Fork 操作，不需要开新树状缩进。连线图案上，采用一个类似 GitHub Branch 抛出 `┣━━▶` 的曲线图形结构并略带透明或变灰色，表明它脱离了主干环境。
3. **卡片结构块 (Right Body)**:
   - 从四行改为**绝对双行展示**（极致两行）。
   - **Row 1**: [图点/状态标记] + 标题 + (可选的右侧小体积徽标如 `[Fork]`/`[Compact]`)
   - **Row 2**: Intent 与最新回复融合的一句单行摘要，并执行裁剪（Overflow ellipsis）。取消所有多余 Hashtags 强占的一行。

### 3.2 节点状态表现
- **Active 节点**：脊索的主节点使用强调色（如亮橙 #fb923c）且自带外发光或粗边款。对应的双行Card 也具有高亮边界。
- **弃用节点 (Abandoned/Forked)**：视觉往回推。采用 `dim` 或者是低透明度，防止非主线工作喧宾夺主。

## 四、 右侧详情抽屉：四大视觉组件

即使只展示选定的详情上下文，传统文本也过于喧闹。新的 Detail Drawer 严格约束所有的呈现内容进入以下“四大职能块”。

### 1. Summary 区块 (玻璃态标题)
- **展示内容**: 会话原生名、Status、分支。
- **视觉风格**: 不再使用表格或传统的 `key: value` 列表。
- **高光组件 - Intent 块**: 最显眼的位置给一句斜体的“意图摘要引用语”。它是这个任务被创立的原生冲动（例如："去解决刚才报错的 TypeError"），左侧伴以加粗色块引导线。

### 2. Breakpoint 区块 (皇冠明珠 MUST DO)
- **展示内容**: 当前的断点（上次结束时的状态或接下来的唯一任务点）。
- **视觉风格**: 这是整个版面最抓取眼球的核心焦点！
  - 背景：带有极其微弱主题色的渐变/阴影容器（如 rgba(251,146,60,0.05)）。
  - 边框：硬边线（或者 Glowing 发光线）。
  - 角标：容器的左上角悬浮一枚极度高反差的 `📍 当前断点 MUST DO`。这样无论眼睛从哪扫过来，必先着陆在该点。

### 3. Milestones 区块 (时序折线 Stepper)
- **展示内容**: 自动/AI提取的会话推进历史记录（里程碑）。
- **视觉风格**:
  - 抛弃黑实的文字列表结构。
  - 使用纵向引导线串连小圆点构成的 Stepper (步骤进度条) 组件。
  - **已完成节点**: 圆点填充为实心灰色带 ✓，并使用 `text-decoration: line-through` 和降噪文本，弱化它们。
  - **进行中节点**: 圆点发亮、文本高亮放大。

### 4. Last Reply 区块 (聊天气泡还原)
- **展示内容**: LLM 提供给用户的最后交互（或者最后抛出的 Diff、代码等）。
- **视觉风格**:
  - 刻意模仿 Chat 软件（保留头像 Avatar 或名字）。
  - 使用具有包裹感的气泡框底色处理，它和上方所有的规范整理组件（如 Stepper 或高亮框）形成了本质的区隔感。它表明这是原生的、无加工的输出文案。

## 五、 将对后续实施的建议

为了在 Textual (Python 系 TUI 框架) 中落地这套界面，技术动作分解建议如下：

1. **Drawer 组件研发**:
   需要在 `tui/screens/` 层级设计一个绝对定位的独立 Widget，使用 `visibility` 或 `display` 辅以层级堆叠管理（Stack / z-index）。
2. **Spine View Canvas 渲染**:
   Textual 中的字符画渲染可以直接通过复写 `render()` 方法时，判断节点的前驱/后驱节点关系。用 `rich.text.Text` 内置连线字符 （ 如 `│`, `╰`, `├` 等）进行硬拼接组装。
3. **Card 的事件监听重构**:
   点击不再是引发右侧子面板内容的单纯换数据动作，它将触发生命周期的 Drawer 显示回调。
